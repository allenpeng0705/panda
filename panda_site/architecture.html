Architecture

A deep dive into Panda's design and components

Overall Architecture

Panda is designed with a four-layer architecture that handles AI traffic efficiently and securely. Each layer has a specific responsibility, allowing for modularity and extensibility.

Four-Layer Design

1. Ingress Layer ("The Mouth")

Accepts high concurrency connections and normalizes inbound protocols into an internal Panda standard representation.

  • Runtime: Async I/O with Tokio, HTTP with Hyper
  • Protocol Support: OpenAI-compatible REST (including streaming), gRPC, MCP-over-SSE
  • Responsibilities: Connection lifecycle, backpressure, timeouts, early request validation

2. Shield Layer ("The Bamboo")

Security and hygiene before expensive or risky downstream work.

  • Prompt-injection firewall: Inspects inbound text for attempts to override system policy
  • PII masking: Detects and redacts sensitive information in near real time
  • Identity exchange: Validates enterprise OIDC/JWTs and maps identities to scopes

3. Intelligence Layer ("The Brain")

Intent, routing, reuse, and tool surfacing.

  • Intent router: Classifies prompts and routes to the best-fit model
  • Semantic cache: Vector similarity over prompts to skip redundant LLM calls
  • MCP aggregator: Maintains connections to multiple MCP servers and exposes a unified tool surface

4. Egress Layer ("The Hands")

Adapts the internal representation to each backend's reality.

  • Universal adapters: Translate Panda's canonical request into vendor-specific formats
  • Stream emulator: Buffers and re-emits full responses as streams
  • Backend quotas: Enforces per-backend limits and transforms errors

Request Lifecycle

  1. Ingress: Client sends an OpenAI-style request (e.g., /v1/chat/completions)
  2. Authentication: JWT/JWKS validation and identity mapping
  3. Policy Checks: Prompt safety, PII scrubbing, Wasm plugin execution
  4. Budget Check: TPM validation and semantic cache lookup
  5. Model Routing: Intent classification and backend selection
  6. Tool Orchestration: MCP tool calls if needed
  7. Upstream Request: Forward to model provider with appropriate adapter
  8. Response Handling: Stream management and error transformation
  9. Observability: Metrics, logs, and traces emission
  10. Client Response: Return OpenAI-shaped response to client

Rust Crates Structure

Panda is organized as a Cargo workspace with multiple crates, each with a specific responsibility:

panda-config

Types, validation, and YAML load for listener address, upstream base URL, and future routes/plugins—no socket I/O.

panda-proxy

Async HTTP server (Tokio + Hyper); streaming reverse proxy; KONG handshake; in-memory TPM hint.

panda-server

Thin binary crate; wires config path (CLI) → panda-config → panda-proxy::run.

panda-wasm

Wasmtime-based plugin host and guest ABI for extensibility.

panda-pdk

Plugin Development Kit for creating Wasm plugins in Rust and Go.

wasm-plugin-sample

Example Wasm plugins demonstrating different use cases.

Design Philosophy

Key Principles

  • Stateless Core: No local database required in the default data plane
  • Stream-native: Built for the LLM "typewriter" effect (SSE and kin)
  • Wasm extensibility: Plugin-first customization without Lua lock-in
  • Intent-aware: Security and routing that reason about meaning, not only paths
  • The layered play: Designed to sit behind or beside KONG-class gateways

Strategic Differentiation

Target Panda Emphasis Typical Gap
Simplicity Single binary, no mandatory control plane database Heavier dependency stacks or embedded script runtimes
Cost control Token-based limits (TPM) + semantic cache Often RPS-oriented metering only
Extensibility Rust + Wasm plugins (including Go support) Historic Lua-centric extension models
Observability Intent- and tool-centric audit logs URL- and status-centric access logs
